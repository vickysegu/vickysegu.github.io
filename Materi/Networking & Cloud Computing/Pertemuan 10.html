Pertemuan 10 · Praktikum Infrastruktur Web

1. Pengantar: Inisialisasi Environment Server Web

Pertemuan 10 berfokus pada penyediaan (provisioning) environment server web dari kondisi kosong (bare-metal virtual machine) hingga siap menyajikan aplikasi web dinamis. Kita akan memasang komponen **HTTP Daemon** (Apache/NGINX) dan PHP Runtime Engine untuk mengubah Virtual Private Server (VPS) menjadi infrastruktur publik yang dapat merespons permintaan klien secara global.

Blok 01 · Jembatan Teori

2. Jembatan Teori: Peran Web Server & Runtime

Memahami pemisahan tanggung jawab antara mesin virtual (hardware terabstraksi) dengan perangkat lunak server web (software pengelola request). Meskipun VPS aman dan dapat diakses secara administratif melalui SSH, server tersebut tetap pasif dan tidak dapat diakses publik sebelum dipasang HTTP daemon untuk mendengarkan port jaringan eksternal.

VPS Teramankan (Pertemuan 9)

Status: Sistem terproteksi. Hanya dapat diakses administrator secara remote via port SSH (22). Layanan sistem berkas masih statis dan tertutup dari akses HTTP.

Sesudah — Pertemuan 10

Server Web Aktif (Pertemuan 10)

Status: Server web siap publik. Layanan latar belakang HTTP Daemon (Apache) berjalan secara persisten, mendengarkan Port 80, dan siap mengeksekusi script PHP via interpreter.

Tanpa Web Server, VPS secanggih apa pun tetap tidak bisa dijadikan website — sama seperti punya rumah bagus tapi tidak ada pintu dan bel. Web Server-lah yang "mendengarkan" di pintu jaringan (port) dan menjawab setiap kunjungan browser. Ini adalah fondasi wajib sebelum kita bicara soal aplikasi web apa pun, baik itu toko online, blog, maupun sistem akademik.

Setelah memahami peran penting web server dan runtime sebagai pondasi operasional VPS, bagaimana cara browser di komputer kita berinteraksi secara teknis dengan server tersebut? Mari kita bedah alur komunikasinya.

Blok 02 · Protokol Transmisi

3. Protokol Transmisi: Arsitektur Request-Response HTTP

HTTP bekerja di atas model transaksi request-response. Browser bertindak sebagai klien yang menginisiasi koneksi TCP ke port web server, mengirimkan berkas permintaan teks (GET/POST), lalu menunggu server memproses dan mengembalikan respons teks terstruktur.

Klien (Browser)
Port 80
Server (Apache)
Memahami alur ini membantu kita saat troubleshooting nanti. Kalau website "tidak bisa dibuka", masalahnya bisa ada di sisi klien (browser), di jalur jaringan (port tertutup), atau di sisi server (servernya mati/belum terpasang). Tanpa paham alurnya, kita akan menebak-nebak dan sulit mencari akar masalahnya.

Komunikasi HTTP yang dinamis di atas hanya dapat terjadi jika browser dan server telah tersambung secara fisik dan logis. Bagaimana protokol TCP menjamin sambungan ini terbentuk dengan aman sebelum pertukaran data dimulai? Mari pelajari proses jabat tangan tiga langkah.

Blok 02 · Lapisan Transportasi

4. Lapisan Transportasi: Protokol TCP Three-Way Handshake

Sebelum data HTTP ditransmisikan, protokol TCP menjamin keandalan sambungan fisik dan logis antara klien dan server melalui mekanisme jabat tangan tiga langkah. Proses ini memvalidasi sinkronisasi nomor urut paket guna mencegah hilangnya data di jaringan.

KlienSYN — "Halo, saya mau connect ke Port 80" →Server
Server← SYN-ACK — "Halo, saya dengar, siap kok"Klien
KlienACK — "Oke, ayo mulai" — koneksi terbentuk →Server
Handshake memastikan kedua belah pihak benar-benar siap sebelum data penting dikirim, sehingga data tidak hilang atau rusak di tengah jalan. Ini juga alasan kenapa membuka website terasa ada jeda sepersekian detik sebelum halaman muncul — di balik layar, ada "sapaan" tiga langkah ini yang terjadi lebih dulu.

Setelah memahami pembentukan koneksi dasar TCP, aturan komunikasi tingkat tinggi (HTTP) memiliki beberapa karakteristik khas yang membuatnya sangat andal di internet. Mari telusuri empat sifat penting protokol HTTP.

Blok 02 · Karakteristik HTTP

5. Karakteristik HTTP: Stateless, Connectionless & Media Independent

Memahami tiga sifat utama protokol HTTP yang mendasari cara kerja aplikasi web modern di internet. Karakteristik ini dirancang untuk menjaga efisiensi penggunaan memori server ketika melayani ribuan kunjungan sekaligus.

📭 Stateless

Server tidak menyimpan status atau riwayat sesi request sebelumnya dari klien yang sama. Setiap transaksi dianggap unik dan berdiri sendiri.

⚡ Connectionless

Koneksi diputus seketika setelah respons dikirimkan. Browser harus menginisiasi jabat tangan TCP baru untuk request berikutnya.

📦 Media Independent

HTTP dapat mengirimkan format data apa pun selama tipe MIME disepakati bersama oleh klien dan server.

✍️ Text Oriented

Seluruh struktur request dan response dikirim dalam format teks murni sehingga mudah dibaca dan didiagnosis.

🧠 Cek Pemahaman: Kenapa website butuh fitur "login tetap tersimpan" padahal HTTP itu stateless?

Dengan pemahaman utuh tentang protokol HTTP, langkah berikutnya adalah memilih dan memasang paket perangkat lunak (web stack) yang sesuai pada server virtual kita. Mari kita bandingkan dua metode instalasi terpopuler.

Blok 03 · Memilih Alat yang Tepat

6. LAMP vs XAMPP — Dua Cara Menyalakan Web Stack

Menganalisis perbedaan arsitektur penyebaran web stack antara metode pemasangan komponen terpisah secara native (LAMP/LEMP) untuk lingkungan produksi yang terisolasi dengan instalasi paket XAMPP praktis untuk mempercepat simulasi dan pengembangan lokal.

Komponen terpasang terpisah & menyatu dengan OS Linux. Keamanan bawaan tinggi (90%), efisiensi RAM maksimal (85%), serta update keamanan otomatis via apt. Standar utama untuk VPS publik.

Karena VPS kita akan diakses lewat internet oleh siapa saja, keamanan jadi prioritas. XAMPP dirancang untuk dipakai di laptop sendiri yang tidak terhubung ke publik — kalau dipasang di server produksi, celah keamanannya (misalnya password kosong di database) bisa dieksploitasi orang lain. LAMP native memang butuh beberapa langkah instalasi tambahan, tapi lebih aman dan lebih hemat sumber daya untuk jangka panjang.

Setelah menimbang kelebihan masing-masing metode instalasi, mari kita praktikkan langsung langkah-langkah instalasi native LAMP stack di terminal VPS kita hingga berhasil ditayangkan.

Blok 03 · Studi Perbandingan

7. Studi Perbandingan: Apache vs Nginx HTTP Daemon

Memilih web server yang tepat memerlukan analisis beban kerja server. Apache menggunakan arsitektur berbasis proses/thread (blocking), sedangkan NGINX menggunakan loop event non-blocking (asynchronous) yang sangat efisien untuk menangani koneksi konkuren dalam jumlah besar.

1. "Situs Tidak Bisa Dijangkau"

Apache & UFW aktif, tapi dari browser gagal terhubung.

http://vps_ip_address
✗ Situs tidak dapat dijangkau — koneksi timeout.
  • ✓Apache berjalan & UFW mengizinkan Port 80
  • ✗Inbound Rule (Firewall Cloud) belum dibuka
Dua lapis firewall harus sinkron: UFW (dalam VPS) dan firewall provider cloud (luar VPS).

2. Kode PHP Muncul Teks Mentah

  • ✗Modul PHP belum aktif di konfigurasi Apache
